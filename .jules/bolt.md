## 2024-05-18 - Sequential Network I/O in Mass Migration

**Learning:** The Firebase RTDB to Cloud Firestore migration function (`_fbMassMigrate` in `src/firebase.js`) was sequentially executing `await get(...)` and `await setDoc(...)` operations within multiple `for...of` loops over collections of books, submissions, and settings. This sequential I/O creates a significant bottleneck, scaling linearly with the size of the database. The same issue was present in the system backup restore function in `src/main.js`.
**Action:** Always replace sequential `await` calls inside loops with `Promise.all()` when the operations are independent (like fetching or saving non-dependent records). This allows the JavaScript runtime to execute the network requests concurrently, drastically reducing the overall latency of the operation.
## 2024-05-19 - Unbounded Promise.all() for I/O

**Learning:** When parallelizing file downloads (like in `syncAllReceipts`), wrapping `Array.prototype.map` inside `await Promise.all()` causes unbounded concurrent network requests. This triggers all requests immediately, which the native browser HTTP layer implicitly throttles to about 6 connections per origin. If a system handles massive datasets (thousands of entries), unbounded `Promise.all` could lead to socket exhaustion or API rate limiting, which wasn't fully apparent on small datasets.
**Action:** When implementing asynchronous I/O batching using `Promise.all()`, be mindful of scale. Consider adding a small concurrency limiter or chunking approach when the mapped array can grow very large in a production environment, rather than blindly executing hundreds of requests simultaneously.
