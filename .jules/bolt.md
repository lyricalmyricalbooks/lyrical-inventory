## 2024-06-29 - Parallel Shippo Transactions Fetch

**Learning:** I noticed that inside a loop the application was awaiting an asynchronous transaction that hit a networking endpoint. This caused an N+1 problem resulting in unnecessary, sequentially processed promises. By keeping optimistic sets (`fetchedIds`, `importedIds`, `existingRefs`) updated temporarily inside the initial loop, we could successfully parallelize network requests with `Promise.all` and then iterate over the results resolving cleanly or reverting the optimistic insertions.
**Action:** Implemented a split loop process in `src/main.js` which gathers validated array of rows, awaits a mapping of `Promise.all` resolving the expenses, and processes the outcome sequentially.
## 2025-02-28 - Backend Mock Email Concurrency Bug Prevention\n **Learning:** Refactoring synchronous read-modify-write file operations to asynchronous `fs.promises` on a shared JSON file introduces race conditions if requests hit concurrently, causing data overwrites. \n **Action:** Applied a promise chain mutex (`mockEmailMutex = mockEmailMutex.then(async () => {...})`) to serialize asynchronous file access for the mock email endpoint and prevented the race condition.

## 2024-07-21 - Replaced O(N) Array.includes with O(1) Set.has in filter loops
**Learning:** Found multiple instances where `Array.prototype.includes()` was used inside `Array.prototype.filter()` loops to check against reference arrays (like `_ocBulkFailedIds`, `selectedIds`, `c.photos`). This creates a quadratic O(N * M) time complexity, which can cause significant UI thread blocking on larger arrays.
**Action:** Replaced these with a `Set` instantiation outside the loop, using `Set.has()` inside the `.filter()` method for O(1) lookups, dropping time complexity to linear O(N + M).
